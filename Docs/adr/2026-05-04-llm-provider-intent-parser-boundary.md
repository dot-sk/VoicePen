# LLM Provider And Intent Parser Boundary

## Context

VoicePen developer mode already has deterministic command rendering from
configured allowlists. Adding an LLM can improve noisy intent recognition, but
letting the model produce commands or provider-specific behavior leak into the
developer mode layer would make privacy, testing, and safety harder to reason
about.

The first supported providers are Ollama and OpenRouter. They have different
HTTP APIs, but both can be treated as structured JSON providers by VoicePen.
The provider capability is expected to support more than developer-mode intent
parsing, so it must live as an independent feature.

## Decision

Separate the system into three layers:

- The standalone LLM feature sends structured JSON prompts to a configured
  backend and returns model text or typed provider/config errors. It owns
  provider config, routing, HTTP request shape, timeout handling, strict schema
  support, and redaction. It does not know about VoicePen command semantics.
- The intent parsing use case owns the prompt template, first-party intent
  registry, active-context allowlist, privacy gates, response validation, and
  confidence policy. It returns typed intent parsing results.
- The command renderer remains deterministic and allowlist-based. It does not
  accept shell commands generated by an LLM.

Require strict structured-output support for OpenRouter. If a selected
OpenRouter model or provider cannot honor `json_schema`, VoicePen reports a
typed provider/config failure instead of falling back to plain JSON prompt
instructions.

## Consequences

Provider code lives under `VoicePen/Features/LLM`, stays reusable for future
structured LLM use cases, and can be tested with HTTP request/response fakes.

The intent parser can evolve its registry and prompt without touching provider
transport code.

Safety remains local: LLM output is an intent classification only, and execution
continues to flow through deterministic allowlists.

OpenRouter compatibility is narrower in v1 because unsupported models fail
explicitly, but the failure mode is predictable and testable.
